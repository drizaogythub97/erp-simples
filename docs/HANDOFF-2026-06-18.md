## 4. Problemas identificados (fila de amanhã)

### 4.1. Bug crítico: link de recuperação de senha cai em `otp_expired`

**Sintoma reportado:** ao clicar no link de "redefinir senha" no e-mail, o
navegador foi para:

```
http://localhost:3000/login?error=N%C3%A3o%20foi%20poss%C3%ADvel%20confirmar%20o%20link.%20Solicite%20um%20novo.#error=access_denied&error_code=otp_expired&error_description=Email+link+is+invalid+or+has+expired&sb=
```

O fragmento `#error=access_denied&error_code=otp_expired` veio do **próprio
servidor de auth do Supabase**, não do nosso código. Acontece principalmente
porque:

1. **Pré-busca de link por scanner anti-phishing do Gmail / antivírus** clica
   na URL antes do usuário, consumindo o OTP. Quando o usuário clica de fato,
   o link já está marcado como usado/expirado.
2. **Templates padrão do Supabase usam `{{ .ConfirmationURL }}`**, que resolve
   para uma URL com o token no *hash fragment* (`#access_token=...`) e fluxo
   *implicit*. Nosso `/auth/callback` só trata `?code=` (PKCE), então quando
   chega como hash, ele nem é processado pelo servidor — fica só no cliente.

**Solução (executar amanhã, nesta ordem):**

1. **Editar `/auth/callback/route.ts`** para aceitar também o padrão
   `?token_hash=xxx&type=recovery&next=/reset` chamando
   `supabase.auth.verifyOtp({ token_hash, type })`. Manter o branch `?code=`
   para PKCE como fallback.

2. **Editar os templates de e-mail do Supabase** (painel → Authentication →
   Email Templates → "Confirm signup", "Magic Link", "Reset Password"). Trocar:

   - `{{ .ConfirmationURL }}` por algo como:
     ```
     {{ .SiteURL }}/auth/callback?token_hash={{ .TokenHash }}&type=signup&next=/dashboard
     ```
     (e `type=recovery&next=/reset` no template de reset)

   - O **assunto** e o **corpo** em pt-BR, com a identidade "ERP Simples".

3. **Pedir ao usuário** que reenvie o link de recuperação **depois** das
   mudanças acima e refaça o passo 13 do smoke test.

### 4.2. Quality-of-life: e-mails em pt-BR e com identidade do sistema

Pedido textual do usuário: "esse e-mail de confirmação seria interessante
personalizá-lo com ao menos o nome do sistema e para que ele fosse enviado em
PT-BR".

**É possível**, no painel do Supabase, em **Authentication → Email Templates**.
Há quatro templates editáveis: Confirm signup, Invite, Magic Link, Reset
Password, Change Email. Para cada um:

- **Subject**: ex. "Confirme seu cadastro no ERP Simples"
- **Body**: HTML em pt-BR, com saudação amigável (público idoso, frase clara,
  link grande), mantendo as variáveis Mustache:
  - `{{ .Email }}`
  - `{{ .SiteURL }}/auth/callback?token_hash={{ .TokenHash }}&type=...&next=...`
  - `{{ .ConfirmationURL }}` *(evitar — não bate com nosso fluxo PKCE/OTP)*

Limitações importantes a comunicar:

- O **From name** ("Supabase Auth") só fica totalmente customizável se o
  usuário **configurar SMTP próprio** (Resend, SendGrid, Mailgun, etc.). É um
  passo extra opcional. Sem SMTP, os e-mails saem do `noreply@mail.app.supabase.io`
  com baixa entregabilidade (cai em spam fácil) — citar isto.
- Supabase não tem i18n nativo por idioma: o template em pt-BR vale pra todos
  os usuários do projeto.

Entregar amanhã: 3 templates prontos (Signup, Magic Link, Reset) em HTML
pt-BR, prontos pra colar no painel.

### 4.3. Quality-of-life: botão "mostrar senha" nos formulários

Pedido textual: "importante incluir mecanismo de opção para ver senha digitada
nos campos tanto de criar conta quanto no de login".

**Implementação amanhã:**

- Criar `components/auth/password-field.tsx`: client component que envolve
  `<Input type="password|text">` com um botão à direita (ícone
  `Eye`/`EyeOff` do `lucide-react`) que alterna o `type` via `useState`.
- Usar em: `/login` (1 campo), `/signup` (1 campo), `/reset` (2 campos).
- Acessibilidade: o botão precisa de `aria-label` ("Mostrar senha" / "Ocultar senha"),
  `aria-pressed`, `type="button"` (pra não submeter o form), e o alvo de
  clique ≥44px.
- Cuidado: não usar `autoComplete` errado ao virar `text` (mantém
  `current-password` ou `new-password` no input).

## 6. Contexto técnico que NÃO PODE esquecer

- **Stack:** Next.js 16.2.9 (App Router, **Turbopack**), React 19.2.4,
  Tailwind v4 (CSS-first via `@theme` em `app/globals.css`), shadcn/ui preset
  `base-nova` (com `@base-ui/react`, **não** Radix), Supabase via
  `@supabase/ssr` 0.12, Zod **4.4.3**.
- **Next 16:** `middleware.ts` foi renomeado para `proxy.ts` e a função
  exportada é `proxy(request)`. Não recriar `middleware.ts`.
- **Zod 4:** Use `z.email("msg")`, não `z.string().email()`. Use
  `z.literal("on", { error: "msg" })`, não `{ errorMap: ... }`.
- **shadcn base-nova:** O Button usa `@base-ui/react/button` que **não**
  suporta `asChild`. Para "link com cara de botão", usar `buttonVariants(...)`
  no `className` do `<Link>`.
- **Tamanhos de input/button:** O default do shadcn é `h-8` (muito pequeno
  para idosos). Em todos os forms de auth aplicamos `h-14 text-lg` no
  `className`. Continuar essa convenção.
- **Regras de segurança (`CLAUDE.md`):** sempre `supabase.auth.getUser()` no
  servidor (nunca `getSession()`); `service_role` só server-side; toda entrada
  validada com Zod no servidor; mensagens de erro pt-BR genéricas.
- **Trigger de signup:** `handle_new_user` em
  `supabase/migrations/0001_init.sql` lê
  `new.raw_user_meta_data ->> 'privacy_accepted'` e grava
  `profiles.privacy_accepted_at` se vier `'true'`. O signup já passa isso
  corretamente em `app/(auth)/signup/actions.ts`.
- **RPC `register_sale(items jsonb)`:** existe e está coberta pelo teste RLS.
  É o caminho oficial para registrar vendas (será usado na Fase 5).
